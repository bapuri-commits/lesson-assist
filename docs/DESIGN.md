## 프로젝트 개요

수업 녹음 파일을 넣으면 **전사 → 요약 → Obsidian 노트**가 자동 생성되는 로컬 CLI 파이프라인.

### 문제

- 90분 강의를 수동 정리하면 30~60분 추가 소요
- 교수님이 판서/슬라이드를 가리키며 설명할 때 음성만으론 맥락이 끊김
- 과제·시험·일정 언급이 강의 중간에 흩어져서 놓치기 쉬움

### 해결

| 단계 | 동작 |
|------|------|
| 입력 | 녹음 파일(m4a/mp3/wav)을 CLI에 전달 |
| 전사 | 로컬 GPU로 faster-whisper 한국어 전사 (타임스탬프 포함) |
| 품질 검토 | 저신뢰 구간을 후보로 추출 → 사용자 교정 기회 제공 |
| 분할 | 긴 전사본을 ~25분 파트로 자동 분할 |
| 요약 | OpenAI API로 파트별 요약 → 통합 요약 생성 |
| 액션 추출 | 과제·시험·일정 언급을 자동 감지 |
| 노트 생성 | Obsidian 마크다운 파일 생성 + The Record 데일리 노트 연동 |

### 환경

- **PC**: Ryzen 9 3900X + RTX 3080 Ti (VRAM 12GB)
- **OS**: Windows 10/11
- **언어**: 100% 한국어, 교수 단독 발화 (화자 구분 불필요)
- **API**: OpenAI (GPT-4o 기본, 품질 최우선)
- **볼트**: Obsidian 로컬 (`G:\CS_Study\The Record`)

### 세 단계 비교

| 구분 | Phase 1: MVP | Phase 2: 확장 | Phase 3: 자동화 |
|------|-------------|---------------|----------------|
| 목적 | 개인 도구 (돌아가는 것) | 포트폴리오 (쓸만한 것) | 포트폴리오 (편한 것) |
| 전사 | faster-whisper + 저신뢰 후보 | + SRT/VTT 출력 | 동일 |

## Phase 1: MVP — 핵심 파이프라인

### 설계 철학

1. **품질 최우선**: 모델은 가장 좋은 것 사용 (whisper large-v3, GPT-4o). 비용보다 정확도
2. **사용자 교정 루프**: 전사가 의심스러운 구간은 반드시 후보로 제시. AI가 "맞다고 확신하는 것만" 최종 결과에 반영
3. **즉시 쓸 수 있는 것**: 설치 후 명령 하나로 노트 생성까지 완료

### 아키텍처

```
[녹음 파일] ─→ [전처리] ─→ [전사] ─→ [품질 검토] ─→ [분할] ─→ [요약] ─→ [노트 생성]
  m4a/mp3/wav    ffmpeg      faster-whisper   저신뢰 후보     ~25분 파트   OpenAI API    Obsidian MD
                 정규화       large-v3 (GPU)   추출/제시       자동 분할    파트별+통합   + 데일리 연동
                                                                          + 액션 추출
```

### 전사 (transcribe)

**엔진**: faster-whisper + large-v3

| 설정 | 값 | 이유 |
|------|----|------|
| model | large-v3 | 한국어 정확도 최우선 |
| language | ko | 자동 감지보다 명시가 안정적 |
| device | cuda | RTX 3080 Ti 활용 |
| compute_type | float16 | VRAM 절약 + 속도 균형 |
| beam_size | 5 | 기본값, 품질 유지 |
| vad_filter | true | 무음 구간 스킵으로 속도 향상 |
| word_timestamps | true | 세그먼트별 타임스탬프 확보 |

**출력물**:
- `{id}_raw.txt` — 전체 전사 텍스트 (plain)
- `{id}_segments.json` — 세그먼트별 텍스트 + 시작/끝 시간 + 신뢰도(avg_logprob, no_speech_prob)

### 품질 검토 (review)

faster-whisper는 세그먼트마다 `avg_logprob`(평균 로그 확률)과 `no_speech_prob`(무음 확률)을 제공한다.
이 값을 기준으로 **저신뢰 구간을 자동 후보화**하고, 사용자가 교정할 수 있도록 제시한다.

**후보 추출 기준** (임계값은 실사용 후 조정):

| 조건 | 기본 임계값 | 의미 |
|------|-------------|------|
| avg_logprob < threshold | -0.7 | 모델이 전사에 자신 없음 |
| no_speech_prob > threshold | 0.5 | 실제 발화가 아닐 가능성 |
| 세그먼트 길이 < 2자 | — | 의미 없는 잡음 전사 가능성 |
| 반복 패턴 | — | 동일 단어/구 3회 이상 반복 (hallucination 징후) |

**교정 워크플로우**:

```
1. 파이프라인 실행 → 전사 완료
2. 저신뢰 후보 목록 출력 (터미널 + review 파일)
   예:
   [1] 35:20-35:45 (logprob: -0.82) "???한 집합에서 ??? 연산을..."
   [2] 41:05-41:12 (logprob: -0.91) "이거를 쪼개면 그 뭐냐..."
   [3] 55:30-55:35 (no_speech: 0.72) "음..."
   
3. 사용자 선택:
   a) 교정 파일 편집 후 재실행 (--review 모드)
   b) 무시하고 진행 (--skip-review)
   c) 대화형 교정 (--interactive)
```

**교정 파일 포맷** (`{id}_review.jsonl`):
```json
{"idx": 1, "start": "35:20", "end": "35:45", "original": "???한 집합에서...", "corrected": "", "action": "pending"}
{"idx": 2, "start": "41:05", "end": "41:12", "original": "이거를 쪼개면...", "corrected": "", "action": "pending"}
```
- `action`: `pending` → 사용자가 `corrected`를 채우고 `accepted`로 변경, 또는 `skipped`

### 분할 (segment)

긴 전사본을 요약 가능한 크기로 나눈다.

**전략**:
- 기본 단위: ~25분 (오디오 시간 기준)
- 분할 지점: 세그먼트 경계 중 **가장 긴 무음 구간**에서 분할 (문장 중간 절단 방지)
- 최소 파트 길이: 5분 (너무 짧은 잔여 파트는 이전 파트에 병합)

**출력물**:
- `parts/part_01.txt`, `part_02.txt`, ...
- 각 파트에 시작/끝 타임스탬프 메타데이터 포함

### 요약 (summarize)

**모델**: GPT-4o (품질 최우선)

#### 1차 요약 — 파트별

각 파트 텍스트를 개별 요약한다.

**프롬프트 출력 구조**:
- 이 파트의 주제 (1줄)
- 목차 (섹션 5~10개)
- 핵심 개념/정의 (정확히)
- 정리/규칙/조건 (있으면 엄밀히)
- 예제/직관
- 교수 강조 포인트 ("시험에 나온다", "중요", "기억해라", "자주 나오는" 등의 신호)
- 헷갈리는 포인트 / 오해 방지
- 과제/읽을거리/다음 예고 (언급된 것만)

#### 2차 요약 — 통합

1차 요약 전체를 합쳐서 하나의 학습 문서로 재구성한다.

**프롬프트 출력 구조**:
- 전체 목차 재구성
- 중복 제거
- 흐름/의존관계 반영 (선행개념 → 후행개념)
- 정의/정리/예제를 재배치하여 "학습용 문서"로 완성

#### 액션 추출 — 과제/시험/일정

요약과 별도로, 전사 전문에서 **행동이 필요한 항목**을 추출한다.

**탐지 대상**:

| 유형 | 키워드/패턴 예시 |
|------|-----------------|
| 과제 | "과제", "레포트", "제출", "다음 주까지", "숙제" |
| 시험 | "시험", "퀴즈", "중간고사", "기말", "범위" |
| 일정 | "다음 주", "휴강", "보강", "변경", "취소" |
| 공지 | "공지", "이클래스", "확인해라", "올려놓을게" |

**출력 포맷** (노트에 삽입 + 데일리 노트 연동):
```markdown
## 액션 아이템
- [ ] **[과제]** 3장 연습문제 5, 7, 12번 — 다음 주 화요일까지 제출
- [ ] **[시험]** 중간고사 범위: 1~5장 — 4월 15일
- [ ] **[일정]** 다음 주 목요일 휴강
```

### 노트 생성 (obsidian_writer)

**출력 경로**: `{vault}/3_Areas/Lectures/{과목명}/YYYY-MM-DD_{과목명}.md`

**마크다운 템플릿**:

## Phase 2: Visual Anchors + RAG + 시험 대비

> Phase 2부터 **포트폴리오**를 겸한다. 코드 품질·문서화·테스트를 강화.

### Visual Anchors

전사 텍스트에서 **시각 자료에 의존하는 구간**을 자동 탐지하고 앵커 체크리스트를 생성한다.

**탐지 키워드**: "칠판", "판서", "보면", "그림", "표", "수식", "식", "여기", "이거", "이렇게", "저기", "슬라이드", "다이어그램", "그래프"

**앵커 후보 생성**:
- 키워드 탐지 시 해당 세그먼트 ±30초 범위를 앵커 후보로 지정
- 주변 전사 텍스트 스니펫(30~60초 분량) 자동 첨부
- 노트의 `## Visual Anchors` 섹션에 체크리스트로 출력

```markdown
## Visual Anchors
### 후보 (자동 생성)
- [ ] **[35:20]** "여기 칠판에 쓴 식을 보면…"
  > 주변 텍스트: "이항 계수를 전개하면... 여기 칠판에 쓴 식을 보면 n choose k가..."
  > 첨부: (없음)
- [ ] **[41:05]** "표로 정리하면…"
  > 주변 텍스트: "이걸 표로 정리하면 시간복잡도가 확 보이는데..."
  > 첨부: (없음)
```

**SRT/VTT 출력**: 전사 결과를 타임스탬프 포함 자막 파일로도 저장. 나중에 녹화 영상과 매칭 가능.

### 강의 컨텍스트 RAG

동일 과목의 이전 강의 요약을 벡터 DB에 저장하고, 새 강의 요약 시 컨텍스트로 활용한다.

**동기**: 교수님이 "지난 시간에 했던 거" 라고 말할 때, 이전 강의 내용을 참조하여 더 정확한 요약 가능.

**설계**:
- 강의 요약 완료 시 자동으로 임베딩 생성 → 로컬 벡터 DB 저장
- 새 강의 요약 프롬프트에 이전 강의 요약 중 관련 부분을 컨텍스트로 주입
- 벡터 DB: ChromaDB (로컬, 설치 간단, 파일 기반)
- 임베딩 모델: OpenAI text-embedding-3-small (또는 로컬 모델 검토)
- 과목별로 컬렉션 분리

**기대 효과**:
- "지난 시간 DFS" → 이전 강의에서 DFS 요약 부분을 자동 참조
- 학기 진행에 따라 누적 컨텍스트 축적 → 요약 품질 점진적 향상
- 강의 간 선수 개념 관계 자동 파악

### 시험 대비 A4

통합 요약을 기반으로 **시험 직전 5분 복습용 압축 문서**를 생성한다.

**프롬프트 출력 구조**:
- 암기할 것 (정의, 공식, 조건)
- 이해할 것 (원리, 흐름, 비교)
- 문제풀이 포인트 (접근법, 유형)
- 자주 나오는 함정 3개
- 5분 초압축 10줄

### eclass_crawler 연동

동일 워크스페이스의 `eclass_crawler` 프로젝트와 **느슨한 결합(Loose Coupling)** 으로 연동한다.
lesson-assist는 eclass_crawler의 출력 디렉토리(`output/`)만 읽으며, 직접 크롤링하지 않는다.

**연동 포인트**:

| 데이터 | 활용 | 구현 |
|--------|------|------|
| 실라버스 주차별 계획 | 날짜 → 주차 주제 자동 파악 → 요약 프롬프트에 주입 | `eclass.py` |
| 다운로드 강의자료 (PDF/PPT) | RAG 컨텍스트로 저장 → 요약 시 참조 | `rag/store.py` |
| 캘린더 이벤트 | 과제/시험 마감일 교차검증 | `eclass.py` |
| 게시판 공지 | 해당 과목 최신 공지 → 노트 첨부 | `eclass.py` |

**설정** (`config.yaml`):
```yaml
eclass:
  enabled: true
  data_dir: "G:\\CS_Study\\eclass_crawler\\output"
  course_mapping:
    머신러닝: "머신러닝"
    데이터베이스: "데이터베이스"
    자료구조: "자료구조"
    객체지향설계와패턴: "객체지향설계와패턴"
    객체지향: "객체지향설계와패턴"       # 단축명
    이산수학: "이산수학"
    공학선형대수학: "공학선형대수학"
    선형대수: "공학선형대수학"           # 단축명
```

`course_mapping`은 lesson-assist에서 사용하는 과목명과 eclass 과목명이 다를 때 매핑한다.
`_load_course`가 `mapped in p.stem` 매칭을 하므로, 기본 과목명은 매핑 없이도 작동한다.
단축명(객체지향, 선형대수)은 편의를 위해 추가.

### 온라인 수업 지원

오프라인/온라인 수업 모두 동일한 파이프라인으로 처리한다.

**입력 흐름**:
- 오프라인: 녹음 파일 (m4a/mp3/wav) → 기존 파이프라인
- 온라인: 화면 녹화 영상 (mp4/mkv/webm) → ffmpeg 오디오 추출 → 기존 파이프라인

**온라인 수업의 추가 이점**:
- 영상에서 Visual Anchor 시점의 **스크린샷 자동 추출** 가능 (판서 촬영 불필요)
- 오디오 품질이 더 좋음 (디지털 음성)

**CLI**:
```bash
# 오프라인 수업 (녹음)
python -m lesson_assist process --audio "lecture.m4a" --course "자료구조"

# 온라인 수업 (영상)
python -m lesson_assist process --input "lecture.mp4" --course "자료구조"
```

### 사용 전략 — 맥락 보전의 3계층

수업 녹음만으로는 시각 자료(판서, 슬라이드 보충)의 맥락이 빠진다.
이 문제를 **자동(RAG) + 반자동(Visual Anchors) + 수동(학생 필기)** 3계층으로 해결한다.

| 계층 | 담당 | 보전하는 맥락 |
|------|------|---------------|
| **1. RAG 자동** | eclass PPT 텍스트 + 이전 강의 요약 | 슬라이드 원본 내용, 이전 강의 선수 개념 |
| **2. Visual Anchors 반자동** | 앵커 탐지 → 학생에게 가이드 | 교수가 PPT 위에 추가한 판서/보충 설명 |
| **3. 학생 수동** | 선택적 사진 촬영 / 간단 메모 | 앵커가 짚어준 구간 중 중요한 것만 보완 |

**핵심 원리**: eclass PPT가 수업의 뼈대이므로 RAG에 넣으면 슬라이드 내용은 자동 참조된다. Visual Anchors는 "PPT에 없는 교수님 추가분"을 짚어주는 역할이다. 학생은 모든 판서를 찍을 필요 없이, 앵커가 표시한 구간 중 핵심만 선택적으로 보완하면 된다.

### 전체 사용 플로우

```
┌─ 수업 전 ──────────────────────────────────────────────┐
│  eclass_crawler 실행 → 강의자료(PPT/PDF) 다운로드      │
│  lesson-assist가 자료 텍스트를 RAG에 자동 저장          │
│  (이전 강의 요약 + PPT 원문이 모두 검색 대상이 됨)     │
└──────────────────────────────────────────────────────────┘
                         ▼
┌─ 수업 중 ──────────────────────────────────────────────┐
│  오프라인: 녹음 + 중요 판서만 선택적 촬영              │
│  온라인:   화면 녹화 (스크린샷 자동 추출됨)            │
└──────────────────────────────────────────────────────────┘
                         ▼
┌─ 수업 후 ──────────────────────────────────────────────┐
│  lesson-assist 실행                                    │
│  → 전사 → 분할 → 각 파트마다 RAG 검색·주입 → 요약     │
│  → Visual Anchors (판서/보충 구간 표시)                │
│  → 노트 생성 → 요약을 RAG에 저장 → 데일리 연동        │
│                                                        │
│  학생: 노트의 앵커 체크리스트 검토                     │
│  → 사진 있으면 첨부 / 없으면 기억·필기로 보완          │
│  → PPT에 있는 내용은 이미 RAG가 처리했으므로 스킵      │
└──────────────────────────────────────────────────────────┘
                         ▼
┌─ 시험 대비 ────────────────────────────────────────────┐
│  lesson-assist exam --course "자료구조"                 │
│  → 누적 노트 기반 A4 1장 압축 문서 생성                │
└──────────────────────────────────────────────────────────┘
```

## Phase 3: 자동화 + 이미지 연결

### Watch 모드

지정 폴더를 감시하여 새 녹음 파일이 들어오면 자동으로 파이프라인을 실행한다.

```bash
python -m lesson_assist watch \
  --inbox "D:\Recordings\inbox" \
  --vault "G:\CS_Study\The Record"
```

- watchdog 라이브러리로 폴더 감시
- 새 오디오 파일 감지 → 파일명에서 과목명 파싱 (규칙: `YYYY-MM-DD_과목명.확장자`)
- 파이프라인 자동 실행
- 교정 단계는 건너뛰거나 나중에 일괄 처리 (교정 대기 큐)

### 이미지 반자동 연결

판서/슬라이드 사진을 Visual Anchors에 자동 연결한다.

**파일명 규칙**:
```
YYYY-MM-DD_과목명_MMmSSs.jpg
예: 2026-03-03_자료구조_35m20s.jpg
```

**동작**:
1. 이미지 inbox 폴더 감시
2. 파일명에서 날짜, 과목명, 시간 파싱
3. 해당 강의 노트의 Visual Anchors에서 가장 가까운 앵커 찾기
4. 앵커에 `![[이미지파일명]]` 자동 삽입

### Resume 기능

파이프라인 실행 중 실패 시 중간 산출물부터 재개한다.

- 각 단계 완료 시 상태 파일(`{id}_state.json`) 저장
- `--resume` 플래그로 마지막 성공 단계 이후부터 재실행
- 상태: `transcribed` → `reviewed` → `segmented` → `summarized` → `written`

### 데일리 노트 자동 연동

강의 노트 생성 시, 해당 날짜의 데일리 노트에 **두 가지**를 자동 추가한다.

#### 1. 공부 기록 섹션

```markdown
## 공부 기록
- 📖 [[2026-03-03_자료구조|자료구조 5강]] — 이진 탐색 트리, AVL 트리
```

#### 2. 액션 아이템 → Todo/일정 섹션

강의에서 추출한 액션 아이템을 데일리 노트의 적절한 섹션에 삽입:

| 액션 유형 | 데일리 노트 섹션 | 예시 |
|-----------|-----------------|------|
| 과제 | `## Todo` | `- [ ] 자료구조 3장 연습문제 5, 7, 12번 (마감: 03/10)` |
| 시험 | `## 일정` 테이블 | `\| 04-15 \| 자료구조 중간고사 (범위: 1~5장) \|` |
| 일정 변경 | `## 일정` 테이블 | `\| 03-10 \| 자료구조 휴강 \|` |

**규칙**:
- 데일리 노트가 없으면 `Templates/daily.md` 기반으로 생성 (Templater 문법 치환)
- 이미 있으면 해당 섹션에 **추가** (기존 내용 유지)
- 중복 방지: 동일 과목+동일 날짜의 항목이 이미 있으면 스킵

## 실전 튜닝 가이드

> 1주일 실사용 후 아래 항목별로 피드백을 기록하고, 그에 따라 파라미터를 조정한다.
> 각 항목에 `[피드백]` 태그로 실전 결과를 적어두면 다음 세션에서 바로 반영 가능.

### 1. Visual Anchors 키워드

**현재 상태**: 수리논술 테스트에서 98개 앵커 생성 — false positive 과다.

**현재 키워드 목록** (`config.py AnchorsConfig`):
```
고정밀 (유지): "칠판", "판서", "슬라이드", "PPT", "피피티", "프레젠테이션", "다이어그램"
중간 (관찰 필요): "그림", "수식", "도표", "그래프", "화면"
문제 (false positive 유발): "보면", "이렇게"
복합 키워드 (현재): "여기 보면", "이거 보면", "저기 보면", "화면에"
```

**튜닝 방향**:

| 변경 | 이유 | 조건 |
|------|------|------|
| `"보면"` 제거 | 한국어 말습관으로 시각자료 무관하게 남발됨 | 실전 데이터에서도 false positive 많으면 즉시 제거 |
| `"이렇게"` 제거 | "이렇게 하면" 등 일반 구어체 | 동일 |
| `"여기 보시면"`, `"화면을 보면"` 추가 | "보면" 대신 더 구체적 복합 패턴 | 실전 전사본에서 교수님이 실제 쓰는 표현 확인 |
| `"필기"`, `"적어"`, `"노트"` 추가 고려 | 판서 상황 직접 언급 | 과목별로 다를 수 있음 |

**과목별 차이 예상**:
- 수학 과목 (선형대수, 이산수학): 판서 비중 높음 → 앵커 많을 수 있음 (정상)
- 이론 과목 (DB, 객체지향): PPT 위주 → 앵커 적을 수 있음
- 실습 과목 (자료구조): 코드 시연 → "화면", "코드", "실행" 키워드 추가 고려

**`merge_gap_seconds` 튜닝**:
- 현재 15초: 15초 이내의 연속 앵커는 하나로 병합
- 너무 짧으면 → 같은 슬라이드 설명 중 앵커가 여러 개
- 너무 길면 → 다른 슬라이드의 앵커가 합쳐짐
- 예상 적정값: 20~30초 (슬라이드 한 장 설명 시간 기준)

**[피드백 기록란]**:
```
과목:
날짜:
앵커 수: 개 (체감 적절 수준: ~개)
주요 false positive 키워드:
추가해야 할 키워드:
merge_gap 체감:
```

### 2. 요약 프롬프트

**현재 상태**: 범용 프롬프트 하나로 모든 과목 처리.

**현재 파트 요약 구조** (`prompts.py PART_SUMMARY_USER`):
```
## 이 파트의 주제
## 목차
## 핵심 개념/정의
## 정리/규칙/조건
## 예제/직관
## 교수 강조 포인트
## 헷갈리는 포인트
## 과제/읽을거리/다음 예고
```

**과목별 튜닝 고려사항**:

| 과목 유형 | 현재 프롬프트의 문제 | 조정 방향 |
|-----------|---------------------|-----------|
| 수학 (선형대수, 이산수학) | "정리/규칙/조건"에 수식이 제대로 전사되지 않을 수 있음 | LaTeX 표기 지시 추가, 증명 흐름 강조 |
| 이론 (머신러닝, DB) | 알고리즘 단계/비교가 핵심인데 구조가 약함 | "알고리즘 단계별 동작" 섹션, "vs 비교" 섹션 추가 |
| 설계 (객체지향) | 디자인 패턴, UML 다이어그램 언급이 많음 | "패턴 구조", "적용 상황" 섹션 |
| 실습 (자료구조) | 코드 구현, 복잡도 분석이 핵심 | "구현 포인트", "시간/공간 복잡도" 섹션 |

**즉시 적용 가능한 개선** (과목 무관):
- 전사 품질 이슈 안내: "전사 과정에서 수식이나 영어 용어가 부정확할 수 있다. 문맥상 올바른 용어로 교정하라."
- eclass 주차 주제 활용 강화: 현재 `topic_hint`를 통합 요약에만 넣는데, **파트 요약에도** 주입하면 요약이 주제에 더 집중할 수 있음

**과목별 프롬프트 분기 구현 방식** (향후):
```yaml
# config.yaml (안)
summarize:
  profiles:
    default: "prompts/default.txt"
    math: "prompts/math.txt"      # 수식, 증명 강조
    theory: "prompts/theory.txt"  # 알고리즘 비교 강조
  course_profile:
    선형대수: math
    이산수학: math
    머신러닝: theory
    데이터베이스: theory
```
→ 1주일 사용 후 "이 과목은 이 섹션이 필요없고 이 섹션이 필요하다"가 확인되면 구현

**[피드백 기록란]**:
```
과목:
날짜:
불필요한 섹션:
빠진 내용 유형:
전사→요약 과정에서 손실된 핵심 내용:
요약 길이 체감 (너무 길다 / 적절 / 너무 짧다):
```

### 3. 오디오 전처리 파라미터

**현재 기본값**:
```yaml
highpass_freq: 80       # Hz
lowpass_freq: 7500      # Hz
denoise_strength: 25    # dB
silence_threshold_db: -40.0
min_silence_duration: 2.0  # 초
```

**튜닝 포인트**:

| 파라미터 | 증상 → 조정 |
|----------|-------------|
| `silence_threshold_db` | 교수님 작은 소리가 잘리면 → -45~-50으로 낮춤 / 잡음이 남으면 → -35로 높임 |
| `min_silence_duration` | 자연스러운 문장 사이 끊김 → 3~5초로 늘림 / 긴 쉬는 시간이 안 잘리면 → 1초로 줄임 |
| `denoise_strength` | 전사에서 없는 말이 생기면 → 15~20으로 줄임 / 잡음이 많으면 → 30~35로 높임 |
| `lowpass_freq` | 교수님 목소리가 뭉개지면 → 8000~10000으로 높임 |
| `highpass_freq` | 에어컨 소리가 남으면 → 100~120으로 높임 |

**전처리 효과 측정 방법**:
1. 같은 녹음을 `--no-clean`과 기본값 둘 다로 처리
2. 비교: 전사 시간, 전사 품질(오탈자), 무음 제거율
3. `CleanStats`의 `removed_seconds`와 `reduction_pct` 확인

**강의실별 차이 예상**:
- 대형 강의실: 반향음 있을 수 있음 → `denoise_strength` 높여야
- 소규모 강의실: 조용 → 기본값 충분
- 녹음기 위치: 교탁 가까이 vs 좌석에서 → 마이크 거리에 따라 `silence_threshold` 조정

**[피드백 기록란]**:
```
과목/강의실:
녹음기 위치:
원본 길이: → 전처리 후:  (제거율: %)
전처리 ON/OFF 전사 품질 차이:
특이사항 (에어컨, 마이크 피드백 등):
```

### 4. 전사 품질

**현재 설정**:
```yaml
transcribe:
  model: large-v3
  beam_size: 5
  vad_filter: true

review:
  logprob_threshold: -0.7    # 이 이하면 저신뢰 후보
  no_speech_threshold: 0.5
  min_segment_chars: 2
  max_repeat_count: 3
```

**관찰 포인트**:
- 전공 용어 오인식: "이진 트리" → "이진 트리" (OK) vs "벡터 공간" → "백터 공간" (교정 필요)
- 영어 용어: "gradient descent" → 한글로 변환? 영어 유지? → whisper가 결정
- 수식 발화: "x 제곱 더하기 y 제곱" → 전사는 맞지만 수식으로 변환은 요약 단계 역할
- 반복 발화: 교수님이 "그니까" "그래서" 반복 → `max_repeat_count` 조정

**`logprob_threshold` 튜닝**:
- 현재 -0.7: 꽤 관대
- 교정 후보가 너무 많으면 → -0.5로 좁힘 (높은 확신만 통과)
- 오류가 그냥 통과하면 → -0.8~-0.9로 넓힘

**[피드백 기록란]**:
```
과목:
전사 시간 (분 녹음 → 분 처리):
주요 오인식 패턴:
저신뢰 후보 수: 개 (체감 적절: ~개)
교정이 필요했던 치명적 오류:
```

### 5. RAG 컨텍스트 품질

**현재 설정**:
```yaml
rag:
  top_k: 5          # 검색 결과 수
  chunk_size: 500   # 청크 크기 (문자)
  chunk_overlap: 50
```

**관찰 포인트**:
- RAG가 주입한 이전 강의 내용이 **실제로 도움이 됐는지** (요약에 선수 개념 반영 여부)
- 주입된 컨텍스트가 **관련 없는 내용**이었는지 (다른 과목 컨텍스트 혼입 등)
- eclass PPT 텍스트가 RAG에 들어갔을 때 요약 품질 변화

**`top_k` 튜닝**:
- 적으면 → 선수 개념 참조 부족
- 많으면 → 프롬프트 토큰 낭비 + 관련 없는 내용 혼입
- 2~3주차부터 데이터 쌓이면 체감 가능

**`chunk_size` 튜닝**:
- 작으면 → 맥락 끊김 (한 개념이 여러 청크로 쪼개짐)
- 크면 → 검색 정밀도 하락
- 예상 적정: 300~700 (과목 특성에 따라)

**[피드백 기록란]**:
```
과목:
n번째 강의:
RAG 주입 여부 (로그 확인):
요약에 이전 강의 참조 반영됨? (Y/N):
관련 없는 컨텍스트 혼입? (Y/N):
eclass 자료 RAG 효과 체감:
```

### 6. 통합 요약 품질

**관찰 포인트**:
- 파트별 요약은 괜찮은데 **통합 요약이 너무 압축**되는지
- 파트 간 **중복 제거가 너무 공격적**인지 (다른 맥락의 유사 내용까지 삭제)
- **선행→후행 재배치**가 잘 되는지 (원래 강의 순서 vs 논리 순서)
- eclass 주차 주제(`topic_hint`)가 통합 요약에 반영되는지

**[피드백 기록란]**:
```
과목:
파트 수:
통합 요약 길이 체감:
빠진 핵심 내용:
잘못 재배치된 내용:
```

## 오디오 전처리 (clean_audio)

전사 전에 오디오를 정리하여 **전사 속도와 품질을 동시에 향상**시킨다.

### 문제

- 강의 녹음에는 수업 시작 전 대기, 쉬는 시간, 교수님 자리 이동 등 긴 무음 구간이 포함
- 강의실 HVAC(냉난방), 옆 교실 소리, 학생 잡음 등 배경 소음
- 녹음 레벨이 일정하지 않음 (교수님이 마이크에서 멀어지는 경우)

### 해결: 3단계 ffmpeg 필터 체인

```
[원본 오디오]
    │
    ├─ 1. 주파수 필터 (highpass 80Hz + lowpass 7500Hz)
    │      → 저주파 럼블(HVAC, 교통) + 고주파 히스 제거
    │
    ├─ 2. FFT 디노이즈 (afftdn, 가용 시)
    │      → 배경 잡음 패턴 학습 후 제거
    │
    ├─ 3. 무음 제거 (silenceremove)
    │      → 2초 이상, -40dB 이하 구간 제거
    │
    ├─ 4. 음량 정규화 (loudnorm EBU R128)
    │      → 일정한 음량으로 표준화
    │
    ▼
[전처리된 오디오] → 전사에 입력
```

### 설정 (`config.yaml`)

```yaml
clean_audio:
  enabled: true           # false면 전처리 건너뜀
  highpass_freq: 80       # Hz, 이 이하 저주파 제거
  lowpass_freq: 7500      # Hz, 이 이상 고주파 제거
  denoise: true           # FFT 디노이즈 (afftdn)
  denoise_strength: 25    # dB, 높을수록 공격적
  remove_silence: true    # 긴 무음 구간 제거
  silence_threshold_db: -40.0  # 이 이하를 무음으로 판단
  min_silence_duration: 2.0    # 초, 이 이상 무음만 제거
  normalize: true         # EBU R128 음량 정규화
```

### CLI

```bash
# 전처리 포함 (기본)
python -m lesson_assist process --audio lecture.m4a --course "자료구조"

# 전처리 끄기
python -m lesson_assist process --audio lecture.m4a --course "자료구조" --no-clean
```

### 파이프라인 위치

```
STEP 0:   영상 → 오디오 추출 (영상 입력 시)
STEP 0.5: 오디오 전처리 ← NEW
STEP 1:   전사 (faster-whisper)
STEP 2:   품질 검토
...
```

### 주의사항

- 전처리 후 **타임스탬프가 원본과 달라진다** (무음 제거 때문). Visual Anchors/자막의 타임스탬프는 전처리된 오디오 기준이 된다. 원본 녹음의 특정 시점을 찾으려면 원본 파일을 참조해야 한다.
- 전처리된 파일은 `data/cleaned_audio/`에 캐시되므로, 같은 파일을 다시 처리할 때 재연산하지 않는다.
- 전처리 실패 시 원본 파일로 자동 폴백한다 (파이프라인이 멈추지 않음).

## 수업자료 처리 통합 방침

### 결정: lesson-assist가 모든 수업자료 처리를 담당

eclass_crawler의 Material Analyzer(설계만 존재)를 lesson-assist에 통합한다.

```
eclass_crawler                    lesson-assist
───────────────                   ─────────────
크롤링 + 다운로드 (여기까지만)  →  텍스트 추출 + RAG 저장
                                   + 독립 노트 생성 (향후)
                                   + 녹음 요약 시 참조
```

**이유**: eclass_crawler는 "데이터 수집" 역할에 집중하고, "분석/요약/노트 생성"은 이미 lesson-assist에 LLM 호출, 프롬프트, Obsidian 연동이 모두 갖추어져 있으므로 여기서 처리하는 게 자연스럽다.

**현재 상태**:
- `material_loader.py` — PDF/PPT 텍스트 추출 → RAG 저장 (구현 완료)
- 독립 노트 생성 — 향후 서브커맨드 `lesson-assist material` 추가 예정

## 노션 연동 (설계 초안)

> 상태: 보류. 필요 시 구현.

### 개요

교수가 노션에 올리는 수업 자료, 과제 설명 등을 크롤링하여 lesson-assist 파이프라인의 RAG 컨텍스트로 활용한다.

### 가능한 연동 포인트

| 데이터 | 활용 |
|--------|------|
| 노션 페이지 텍스트 | RAG 컨텍스트 (수업자료 보충) |
| 과제 설명 | 액션 아이템 교차검증 |
| 수업 일정표 | 주차 매칭 보조 |

### 기술 접근

- **Notion API** (공식): 학생 계정으로 공유된 페이지 접근 — API 키 필요, 교수가 공유 설정해야 함
- **웹 크롤링**: 공개 노션 페이지를 직접 파싱 — 로그인 불필요, 구조 변경에 취약
- **Notion export**: 수동 내보내기 → 마크다운 → lesson-assist 처리 — 가장 단순하지만 수동

### 보류 사유

현재 eclass에서 대부분의 수업 정보(실라버스, 자료, 공지, 캘린더)를 이미 확보 중. 노션을 병행하는 교수가 있으면 그때 구현.
